# 如何创作一个 Reasonix Skill

本教程从零开始，带你完成一个高质量 Skill 的设计、编写、测试和发布全流程。

---

## 1. Skill 是什么

一个 Skill 就是一个 **markdown 文件**，放在 `.reasonix/skills/` 目录下。它包含：

- **frontmatter**（YAML 头部）——元数据：名称、描述、运行模式
- **body**（markdown 体）——给 AI 的指令：做什么、怎么做、有什么规则

Skill 通过 `/skill <name>` 手动调用，或由 AI 根据上下文自动选择。

### 一个最简 Skill

```markdown
---
name: hello-world
description: 打个招呼，输出当前时间。
---
# hello-world

向用户打招呼，并告诉他当前的日期和时间。

## 规则
- 用用户的系统语言回复
- 如果用户有设置名字，带上名字
```

就这么简单。但要做**好**，需要多考虑几步。

---

## 2. 设计一个好的 Skill

### 第一步：明确问题

问自己三个问题：

| 问题 | 好的答案 | 不好的答案 |
|---|---|---|
| 解决什么痛点？ | "每次手动整理下载文件夹很烦" | "一个文件管理工具" |
| 谁会用？ | "经常下载文件的开发者" | "所有人" |
| 频繁吗？ | "每周至少 3 次" | "偶尔" |

### 第二步：确定操作边界

一个 Skill 应该做**一件事**，做清楚。如果发现自己在写 "支持以下 8 种操作"，考虑拆成多个 Skill。

**好的边界：**
- ✅ `text-polish`：润色 / 翻译 / 总结 / 改写——都是「文本处理」
- ✅ `file-organizer`：分类 / 去重 / 目录树 / 清理空目录——都是「文件整理」

**不好的边界：**
- ❌ 「项目管理全能工具」——太宽，AI 不知道什么时候该用

### 第三步：写出清晰的指令

AI 读你的 Skill 就像新员工读操作手册。写得越具体，执行越准确。

**好的指令：**
```markdown
### 去重
- 对比文件名 + 文件大小
- 标记疑似重复（不自动删除）
- 列出建议表：保留哪个、删除哪个、理由
```

**不好的指令：**
```markdown
### 去重
帮我去重文件
```

## 3. Skill 的结构模板

```markdown
---
name: my-skill              # 必填：字母/数字/-/_，唯一
description: 一句话描述       # 必填：≤120 字，会显示在 skill 列表中
runAs: subagent              # 可选：inline（默认）| subagent
---

# my-skill

## 概述
[2-3 句说明这个 Skill 做什么、什么时候用]

## 操作

### 操作 A
- 步骤 1
- 步骤 2
- 输出格式

### 操作 B
- ...

## 规则
- [约束 1：什么能做]
- [约束 2：什么不能做]
- [约束 3：边界条件]

## 示例（可选）
[输入 → 预期输出的例子]
```

### runAs 的选择

| 模式 | 何时用 | 优点 | 缺点 |
|---|---|---|---|
| `inline`（默认） | 简单、快速的操作 | 共享父会话上下文，反应快 | 占用 token |
| `subagent` | 需要大量工具调用、搜索、长时间运行 | 不占主会话上下文 | 无上下文，指令必须自包含 |

**经验法则**：如果 Skill 需要做 web_search + web_fetch（如 learn-topic），用 subagent。如果只是文本处理，用 inline。

## 4. 写好规则（最重要的部分）

规则决定了 AI 的行为边界。三层规则结构：

### 安全规则（硬约束）
```markdown
- 所有破坏性操作**先展示计划，经确认后执行**
- 不操作隐藏文件和系统目录
- 不修改 .git/ 下的任何文件
```

### 行为规则（软约束）
```markdown
- 优先使用中文回复
- 代码块标注语言
- 输出控制在 500 字以内
```

### 质量标准（输出约束）
```markdown
- 改写时标注风格变化
- 代码块原样保留不翻译
- 不确定的信息标注"待验证"
```

## 5. 测试你的 Skill

### 本地测试流程

```bash
# 1. 把 skill 文件放到项目 skills 目录
cp my-skill.md your-project/.reasonix/skills/

# 2. 在 Reasonix 会话中手动调用
/skill my-skill

# 3. 测试各种输入场景
#    - 正常输入
#    - 边界输入（空文件、超大文件、特殊字符）
#    - 错误输入（不支持的格式）
```

### 检查清单

- [ ] Skill 能被 `/skill <name>` 正确调用
- [ ] AI 理解了所有操作模式
- [ ] 破坏性操作有确认步骤
- [ ] 输出格式符合预期
- [ ] 不会无限循环或超长输出
- [ ] 边界情况有合理的错误处理

## 6. 发布到 awesome-reasonix

### 选项 A：直接 PR 到 awesome-reasonix（推荐新手）

```bash
# 1. Fork awesome-reasonix
# 2. 把你的 skill 文件放到 skills/ 目录
# 3. 在 README.md 对应分类下添加条目
# 4. 提 PR
```

### 选项 B：创建独立仓库（推荐成熟项目）

```bash
# 1. 创建独立 GitHub 仓库
# 2. 包含 README.md + skill.md + LICENSE
# 3. 向 awesome-reasonix 提 PR，添加链接
```

**何时用 B？** 当你的 Skill 足够复杂，需要独立 issue 追踪、版本管理、Star 积累时。Claude Code 社区的 Superpowers（87K Star）就是独立仓库。

### PR 格式

```markdown
- **[my-skill](链接)** — 一句话描述。为什么值得收录：关键特性或独特价值。*补充信息：作者、子命令数量等。*
```

参考 [CONTRIBUTING.md](CONTRIBUTING.md) 获取完整规范。

## 7. 常见模式 & 灵感的来源

### 模式 1：操作指南型
- AI 按步骤引导用户完成一个复杂操作
- 例子：数据库迁移、环境搭建、部署流程
- 关键：步骤清晰，有确认点

### 模式 2：自动化工具型
- AI 自动执行一系列操作，输出结果
- 例子：代码审查、测试生成、changelog 生成
- 关键：输入明确，输出格式固定

### 模式 3：知识检索型
- AI 搜索 + 抓取 + 整理信息
- 例子：技术调研、API 文档查询、竞品分析
- 关键：用 subagent 模式，控制输出长度

### 模式 4：转换器型
- 输入 A 格式，输出 B 格式
- 例子：文本润色、代码翻译（Python → TypeScript）、数据格式转换
- 关键：保持原意不添加，标注改动

## 8. 常见错误

| 错误 | 问题 | 修正 |
|---|---|---|
| 指令太模糊 | "帮我整理文件" | "按扩展名归类到 images/、docs/、code/ 等子目录" |
| 没有安全规则 | AI 可能误删文件 | 加"先展示计划，经确认后执行" |
| 边界太宽 | AI 不知道什么时候用 | 明确触发条件和适用范围 |
| 输出无限制 | 可能输出 3000 字 | 加"控制在 500 字以内" |
| frontmatter 缺失 | Skill 无法被识别 | 确保有 `name` 和 `description` |
